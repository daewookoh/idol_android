# 스크린 문서 작성 가이드

> 모든 스크린에 대해 통일된 형태의 문서를 작성하기 위한 가이드입니다.

---

## 📋 문서 작성 원칙

### 목적
* 개발자가 스크린의 비즈니스 로직을 빠르게 이해
* 코드를 읽지 않고도 화면 동작 파악 가능
* API 호출과 데이터 플로우 명확히 문서화

### 작성 대상
* 각 스크린마다 1개의 문서 (스크린 파일과 같은 디렉토리)
* 파일명: `[스크린명].md` (예: `StartUpScreen.md`)

---

## 📄 문서 템플릿

```markdown
# [스크린명]

> 화면에 대한 간단한 설명 (1줄)

---

## 🎯 주요 기능

- 기능 1 간략 설명
- 기능 2 간략 설명
- 기능 3 간략 설명

---

## 📦 UseCases

* **UseCase명1**: 설명 (예: 앱 전역 설정)
* **UseCase명2**: 설명 (예: 사용자 프로필 정보)
* **UseCase명3**: 설명 (예: 타임존 업데이트)

---

## 🌐 호출 API 및 사용 Field

### [그룹명] API
* **GET /api/endpoint1**
  * field1, field2, field3
  * field4, field5

* **POST /api/endpoint2**
  * requestField1, requestField2

* **PUT /api/endpoint3** (설명 추가 가능)
  * field (request body)

## 🔄 화면 플로우

### 1️⃣ 진입
* Navigation: 어디서 진입하는지
* 파라미터: 받는 파라미터 (없으면 생략)
* 조건: 진입 조건 (있는 경우만 작성)

### 2️⃣ 사용자 액션

| 액션 | 설명 | 결과 |
|------|------|------|
| 버튼 클릭 | 어떤 버튼인지 | 어떤 화면으로 이동 |
| 입력 | 입력 필드 설명 | 검증 수행 |
| 스크롤 | 목록 스크롤 | 페이지네이션 |

### 3️⃣ 종료
* ✅ 성공 시 → 다음 화면명
* ❌ 취소/실패 시 → 이전 화면 또는 처리 방법
```

## ✅ 작성 규칙

### ✔️ 반드시 포함해야 할 내용

1. **사용자 관점**
   * 사용자가 할 수 있는 모든 액션
   * 화면 진입/종료 조건
   * 에러 발생 시 사용자가 보는 것

2. **API 정보**
   * 모든 API 엔드포인트 (빠짐없이)
   * 실제 사용되는 필드만 정확히 기록
   * HTTP Method 명시

3. **데이터 흐름**
   * DataStore 저장/읽기 데이터
   * Room DB 테이블 (사용하는 경우)
   * 데이터가 어떻게 흐르는지

4. **비즈니스 로직**
   * 화면의 주요 정책과 규칙
   * 에러 처리 방법
   * 성능 최적화 기법

### ❌ 절대 포함하지 말아야 할 내용

1. **코드 레벨**
   * Kotlin/Java 코드 스니펫
   * 함수명, 클래스명
   * State/Intent/Effect 정의
   * 파일 경로, 라인 번호

2. **구현 세부사항**
   * 어떤 변수를 사용하는지
   * 어떤 함수를 호출하는지
   * ViewModel 내부 구현

3. **추측성 내용**
   * 코드에서 확인되지 않은 내용
   * "아마도", "~인 것 같다" 같은 표현
   * 미래 계획이나 예정 사항

## 📌 섹션별 작성 가이드

### 🎯 주요 기능
* 3-5개의 핵심 기능만 나열
* 각 1줄로 간결하게 작성
* 기술적 용어보다는 일반 용어 사용

### 📦 UseCases
* 화면에서 사용하는 모든 UseCase 나열
* 괄호 안에 간단한 설명 추가
* 미사용 UseCase는 "(미사용)" 표시

### 🌐 호출 API 및 사용 Field
* **가장 중요한 섹션**
* API를 기능별로 그룹화
* 실제 코드에서 사용하는 필드만 기록
* 중첩 필드는 `.`으로 표시 (예: `data.username`)

### 🔄 화면 플로우
* 진입 → 액션 → 종료 순서로 작성
* 사용자 액션 테이블은 주요 액션만 포함
* 복잡한 플로우는 단계별로 나누기

### 💾 데이터 저장
* DataStore: 키 이름만 나열
* Room DB: 테이블 목적과 주요 필드만
* 타입 정보는 필요시에만 추가

### 🎨 UI 구성
* 디자인 스펙이 중요한 경우만 작성
* 테마별 색상 코드 명시
* 크기는 `dp` 단위 사용

### ⚠️ 에러 처리
* 실제 처리되는 에러만 기록
* 에러 메시지는 사용자가 보는 그대로
* 처리 방법 명확히 기술

### ⚡ 성능 최적화
* 적용된 최적화만 작성
* 왜 최적화가 필요한지 간단히 설명
* 없으면 섹션 생략 가능

### 📝 주의사항
* 개발자가 알아야 할 중요 정보
* 예상 시간, 제한사항 등
* 아이콘으로 구분하여 가독성 향상

## 🎨 포맷 가이드

### 이모지 사용
* 각 섹션마다 관련 이모지 사용
* 일관성 유지 (같은 내용은 같은 이모지)
* 표준 이모지:
  * 🎯 주요 기능
  * 📦 UseCases
  * 🌐 API
  * 🔄 플로우
  * 💾 데이터
  * 🎨 UI
  * ⚠️ 에러
  * ⚡ 성능
  * 📝 주의사항

### 테이블 사용
* 정보가 구조화되어 있으면 테이블 사용
* 헤더는 명확하게 작성
* 열 개수는 2-4개가 적당

### 구분선 사용
* 섹션 구분은 `---` 사용
* 과도한 구분선 사용 금지
* 한 섹션이 끝나고 다음 섹션 시작 전에만

### 강조 표시
* **굵게**: 중요 키워드
* `코드`: API, 필드명, 키 이름
* ✅❌ 체크: 성공/실패 상태
* ⚠️: 경고, 중요 사항
